# PROMPT สำหรับ Replit AI Agent: สร้างระบบ AI Multi-Agent SaaS Growth & Support System

คุณคือ Senior Full-Stack Engineer ที่เชี่ยวชาญ Python, FastAPI, PostgreSQL และ AI Agent Framework ให้สร้างระบบเว็บแอปพลิเคชันตามสเปกด้านล่างนี้แบบ **ใช้งานได้จริง (Production-Ready)** ไม่ใช่ Demo หรือ Mock-up

---

## 0. บริบทโปรเจกต์

นี่คือระบบ Virtual Office ส่วนตัวสำหรับผู้ใช้คนเดียว (Founder/CTO) ทำหน้าที่แทนทีมการตลาด-เซลส์-ซัพพอร์ต ให้กับ SaaS ร้านบิวตี้ (ทำเล็บ/ต่อขนตา) เป้าหมายคือหา 100 ร้านค้าแรก ระบบต้อง:
- ทำงานอัตโนมัติจริง ไม่ใช่แค่ตอบคำถามลอยๆ
- แต่ละ Agent มีหน้าที่ชัดเจน ไม่ก้าวก่ายกัน ไม่ "เถียงกัน" หรือ loop ไม่จบ
- อิงข้อเท็จจริงทางการตลาด ไม่แต่งข้อมูลขึ้นเอง (ห้าม hallucinate ตัวเลข/เคส)
- มีกลไกค้นคว้าข้อมูลจริงก่อนสร้าง Output (research-before-generate)
- มีกลไกเรียนรู้/ปรับปรุงตัวเองจากผลลัพธ์ที่ผ่านมา
- **คุมการยิง API เข้มงวด** เพราะ GitHub Models Token มี Rate Limit ถ้ายิงถี่เกินจะโดนตัดการเชื่อมต่อ (bot ต้อง "หลับ" แทนที่จะพัง)

## 1. Tech Stack (บังคับ)

| ส่วนประกอบ | เทคโนโลยี |
|---|---|
| Environment | Replit |
| Deployment | Render (Microservice) |
| Database | Neon DB (PostgreSQL) |
| LLM Provider | GitHub Models (ผ่าน GitHub Token API) |
| Agent Framework | LangGraph (Python) — เลือกใช้แทน CrewAI เพราะควบคุม State Machine และ Rate Limit ได้ละเอียดกว่า |
| Omni-channel Hub | Chatwoot (Webhook รับ Facebook Messenger, Line, Email) |
| Backend API | FastAPI |
| Task Queue / Scheduler | APScheduler หรือ Celery + Redis (สำหรับ Follow-up ตามกำหนดเวลา และหน่วง Retry) |

## 2. Database Schema (Neon DB / PostgreSQL)

สร้างตารางต่อไปนี้ พร้อม migration script (ใช้ Alembic):

```sql
CREATE TYPE lead_status AS ENUM (
  'New', 'Contacted', 'FollowUp_1', 'FollowUp_2', 'Trial', 'Ghosted', 'Blocked'
);

CREATE TABLE leads (
  shop_id SERIAL PRIMARY KEY,
  shop_name VARCHAR(255) NOT NULL,
  facebook_url VARCHAR(500),
  line_id VARCHAR(255),
  status lead_status DEFAULT 'New',
  pain_points JSONB,               -- ผลวิเคราะห์จาก Agent 1
  last_contacted_date TIMESTAMP,
  next_followup_date TIMESTAMP,    -- สำหรับ scheduler เช็คว่าใครถึงคิว
  conversation_history JSONB DEFAULT '[]',
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Log การยิง API ทุกครั้ง เพื่อคุม Rate Limit และ debug
CREATE TABLE api_usage_log (
  id SERIAL PRIMARY KEY,
  agent_name VARCHAR(100),
  model_used VARCHAR(100),
  tokens_used INT,
  status VARCHAR(50),              -- success / rate_limited / error
  created_at TIMESTAMP DEFAULT NOW()
);

-- คิวงานที่รอ Founder อนุมัติ
CREATE TABLE pending_approvals (
  id SERIAL PRIMARY KEY,
  shop_id INT REFERENCES leads(shop_id),
  agent_name VARCHAR(100),
  draft_message TEXT,
  reasoning TEXT,                  -- เหตุผลที่ AI เลือกส่งข้อความนี้ ให้ Founder ตัดสินใจง่ายขึ้น
  status VARCHAR(50) DEFAULT 'pending',  -- pending / approved / rejected / edited
  created_at TIMESTAMP DEFAULT NOW(),
  reviewed_at TIMESTAMP
);

-- เก็บ feedback เพื่อให้ระบบเรียนรู้/ปรับปรุงตัวเอง
CREATE TABLE agent_feedback (
  id SERIAL PRIMARY KEY,
  approval_id INT REFERENCES pending_approvals(id),
  outcome VARCHAR(50),             -- replied / ignored / converted / blocked
  founder_note TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Cache ผลการค้นคว้าข้อมูล (ลดการยิง API ซ้ำ)
CREATE TABLE research_cache (
  id SERIAL PRIMARY KEY,
  query_hash VARCHAR(64) UNIQUE,
  query_text TEXT,
  result JSONB,
  expires_at TIMESTAMP
);
```

## 3. Agent Architecture (LangGraph State Machine)

สร้างเป็น Directed Graph ที่มี Supervisor เป็นตัวกลาง route งาน ไม่ใช่ Agent คุยกันเองแบบอิสระ (ป้องกันการเถียงกัน/loop):

```
Webhook (Chatwoot) → Supervisor Agent → [เลือก Agent ที่เหมาะสม] → Output Validator → 
  → ถ้าเป็นข้อความทั่วไป → ส่งอัตโนมัติผ่าน Chatwoot API
  → ถ้าเป็นข้อความขาย/เคสยาก → บันทึกลง pending_approvals → รอ Founder Approve
```

### Agent 1: Lead Scraper & Analyst
- **หน้าที่:** วิเคราะห์เพจร้านบิวตี้ หา Pain Point จริง (คิวหลุด, ตอบช้า, รีวิวลูกค้าบ่น ฯลฯ)
- **บังคับ research-first:** ต้องดึงข้อมูลจริงจากเพจ/รีวิวก่อนสรุป Pain Point ห้ามเดาเอง ถ้าหาข้อมูลไม่ได้ให้บันทึกสถานะ `insufficient_data` แทนการแต่งขึ้น
- **Output:** บันทึกลง `leads.pain_points` เป็น JSON พร้อมแหล่งอ้างอิง (source URL/timestamp)

### Agent 2: Strategic Closer
- **หน้าที่:** ร่างข้อความขาย/ติดตามตาม Follow-up Logic
- **Follow-up Logic (บังคับตามตาราง):**

| วัน | เงื่อนไข | Action |
|---|---|---|
| Day 1 | ทักครั้งแรก | ทักทายสุภาพ เสนอวิธีแก้ปัญหาฟรี ห้ามขายตรง |
| Day 4 | ถ้าเงียบ | ส่ง Case Study สั้นๆ ของร้านที่ใช้แล้วได้ผลจริง (ต้องมาจาก research_cache ที่ verify แล้วเท่านั้น) |
| Day 7 | ถ้าเงียบ | ส่งลิงก์ Trial พิเศษ ดึงเข้า Self-Service Onboarding |
| เกิน Day 7 | ถ้ายังเงียบ | เปลี่ยนสถานะเป็น `Ghosted` หยุดส่งทันที ห้าม retry เอง |
- ทุกข้อความที่ Agent 2 ร่าง **ต้องผ่าน pending_approvals เสมอ** ห้ามส่งตรงโดยไม่ผ่าน Founder

### Agent 3: Support & Interactive Guide
- **หน้าที่:** ตอบคำถามเทคนิคจาก Knowledge Base เท่านั้น (RAG จาก docs ที่กำหนดไว้ล่วงหน้า ห้ามตอบจากความจำ/เดา)
- ถ้าเจอคำถามที่ไม่มีใน Knowledge Base หรือดูเป็นบั๊กซับซ้อน → สรุปเป็น bullet point → เปิด Ticket ส่งเข้า Admin แทนการเดาคำตอบ
- ข้อความตอบทั่วไป (มีคำตอบชัดเจนใน KB) → ส่งอัตโนมัติได้ทันทีไม่ต้องรอ approve

### Agent 4: Supervisor Agent
- **หน้าที่:** Route งานเข้า Agent ที่ถูกต้อง, ตรวจ Output ทุกตัวก่อนปล่อยออก (เช็คโทนภาษา, ความถูกต้องของข้อมูล, ไม่ผิดกฎ Follow-up)
- ถ้า Agent ใดให้ผลลัพธ์ที่ขัดแย้งกันเอง (เช่น Agent 2 กับ Agent 3 ตอบเรื่องเดียวกันคนละแบบ) Supervisor เป็นผู้ตัดสินสุดท้ายเสมอ ไม่ปล่อยให้ Agent เถียงกันเอง

## 4. กลไกคุม Rate Limit ของ GitHub Models (สำคัญที่สุด — ห้ามข้าม)

ต้อง implement ทุกข้อนี้:

1. **Token Bucket / Sliding Window Limiter** — เขียน middleware ที่คุมจำนวน request ต่อนาทีตาม Rate Limit จริงของ GitHub Models tier ที่ใช้ (ให้ระบบอ่านค่าจาก config `.env` เช่น `MAX_REQUESTS_PER_MINUTE`)
2. **Exponential Backoff + Sleep Mode** — ถ้าโดน 429 (Rate Limited) ให้ bot เข้าสถานะ "หลับ" (sleep) ตาม header `Retry-After` ที่ API ส่งกลับมา ไม่ยิงซ้ำทันที และบันทึกลง `api_usage_log` ว่า `status = rate_limited`
3. **Request Queue** — งานทั้งหมดที่ต้องยิง LLM ต้องเข้าคิวผ่าน Task Queue (ไม่ยิงพร้อมกันหลาย Agent ในเวลาเดียวกัน) จำกัด concurrency ไว้ที่ 1-2 request พร้อมกันสูงสุด
4. **Research Cache** — ก่อนยิง API เพื่อค้นคว้าข้อมูลซ้ำเรื่องเดิม ให้เช็ค `research_cache` ก่อนเสมอ (TTL ตั้งได้ เช่น 7 วัน) เพื่อลดจำนวนการยิง
5. **Batching Follow-up** — งาน Follow-up ที่ scheduler รันทุกวัน ให้ทยอยยิงทีละรายการแบบมี delay คั่น ไม่ยิงพร้อมกันทั้งหมดในครั้งเดียว
6. **Dashboard แสดงสถานะ Token** — หน้า Admin ต้องมี indicator แสดงว่าตอนนี้ bot อยู่สถานะ Active/Sleeping และเหลือ quota ประมาณเท่าไหร่ (ประมาณจาก `api_usage_log`)

## 5. กลไก Research-First (ห้ามให้ AI มั่วข้อมูลการตลาด)

- ทุก Agent ที่ต้องอ้างอิงข้อมูลการตลาด/เคสลูกค้า/สถิติ ต้องผ่านขั้นตอนค้นหาข้อมูลจริงก่อน (web search / ดึงจากฐานข้อมูลภายในที่ verify แล้ว) แล้วค่อยส่งเข้า LLM เพื่อสรุป
- ถ้าไม่มีข้อมูลรองรับ ให้ Agent ตอบแบบ "ไม่มีข้อมูลยืนยัน" หรือ fallback เป็นข้อความทั่วไปที่ไม่ระบุตัวเลข/เคสเฉพาะ แทนการสร้างข้อมูลปลอม
- เก็บ source ของทุกข้อมูลที่ใช้ไว้ใน `research_cache.result` เพื่อ Founder ตรวจสอบย้อนหลังได้

## 6. กลไก Self-Improvement

- หลังจาก Founder Approve/Reject/Edit ข้อความใน `pending_approvals` ให้บันทึกผลลง `agent_feedback`
- ทุกสัปดาห์ ให้มี job สรุปว่าข้อความแบบไหน (โทน, ความยาว, จังหวะ) ได้ผลตอบรับดี (replied/converted) vs แบบไหนถูก reject/ignore บ่อย แล้วสรุปเป็น "แนวทางที่ควรปรับ" ให้ Founder ดูในหน้า Dashboard
- **ห้าม** ให้ระบบแก้ prompt ของ Agent เองอัตโนมัติโดย Founder ไม่รับรู้ — การปรับปรุงต้องเสนอเป็นคำแนะนำให้ Founder กด Apply เอง เพื่อป้องกัน Agent เพี้ยนไปเองแบบควบคุมไม่ได้

## 7. Admin Dashboard (หน้าเว็บที่ Founder ใช้งานจริง)

ต้องมีหน้าเหล่านี้:
1. **Pending Approvals** — list ข้อความรอ approve พร้อมปุ่ม Approve/Reject/Edit และแสดง reasoning ของ AI
2. **Leads Overview** — ตาราง leads ทั้งหมด filter ตาม status
3. **System Health** — สถานะ Token/Rate Limit, จำนวน request วันนี้, sleep mode หรือไม่
4. **Weekly Insights** — สรุปผลจาก Self-Improvement mechanism
5. **Knowledge Base Manager** — สำหรับ Agent 3 ใช้ตอบลูกค้า แก้ไขได้จากหน้านี้

---

## 8. ข้อกำหนดสำคัญที่ห้ามข้าม (Non-negotiable)

- ห้าม Agent ส่งข้อความขาย/follow-up ออกไปโดยไม่ผ่าน Approval (ยกเว้นคำตอบซัพพอร์ตทั่วไปที่มีใน KB ชัดเจน)
- ห้ามยิง LLM API แบบไม่ผ่าน Rate Limiter ไม่ว่ากรณีใด
- ทุกข้อมูลการตลาด/เคสลูกค้าที่ใช้ในข้อความต้องมี source อ้างอิงเก็บไว้ตรวจสอบได้
- Log ทุก error และการเข้า sleep mode ไว้ใน `api_usage_log` เพื่อ debug ย้อนหลัง
- เขียน `.env.example` ให้ครบทุกตัวแปรที่ต้องตั้งค่า (DB connection, GitHub Token, Chatwoot API key, Rate limit config)
