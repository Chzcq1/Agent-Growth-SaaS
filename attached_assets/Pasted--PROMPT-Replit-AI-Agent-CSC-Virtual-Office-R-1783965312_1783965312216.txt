# PROMPT (ฉบับแก้ไข) สำหรับ Replit AI Agent: ปรับโครงสร้างระบบ CSC Virtual Office

> **หมายเหตุถึง Replit Agent:** พร็อมก่อนหน้านี้ทำให้เข้าใจผิดว่าต้องสร้าง Multi-page Admin Dashboard (มีหน้า Pending Approvals, Leads Overview, System Health ฯลฯ แยกกัน) **ให้ยกเลิกแนวคิดนั้น** และสร้างใหม่ตามสเปกฉบับนี้แทน สิ่งที่ต้องการจริงๆ คือ **Virtual Office ที่มี AI เป็นลูกน้อง** ไม่ใช่ระบบ CRM ที่มีหลายหน้าให้ไล่ดู

---

## 0. Concept หลักที่ต้องเข้าใจให้ตรง

Founder จะ **แปะข้อมูลดิบก้อนเดียว** เข้าไปในช่องเดียว (เช่น โพสต์จากกลุ่ม Facebook, คอมเมนต์ลูกค้า, บทสนทนากับลีด, feedback จากร้านที่ใช้อยู่) แล้วให้ระบบ:

1. **แจกจ่ายงานเองภายใน** — Supervisor Agent อ่านข้อมูลที่แปะเข้ามา แล้วตัดสินใจเองว่าควรส่งต่อให้ Agent ไหนวิเคราะห์ส่วนไหน (ไม่ต้อง Founder มานั่งสั่งทีละ Agent)
2. **แต่ละ Agent คิด วิเคราะห์ ทำงานในหน้าที่ของตัวเอง** พร้อมกัน (หรือเรียงตามลำดับที่ Supervisor จัดคิว)
3. **สรุปผลกลับมาเป็นคำตอบเดียว** ในรูปแบบ:
   - **ข้อมูลสำคัญที่พบ** (Key Findings) — สิ่งที่แต่ละ Agent เจอจากข้อมูลที่ Founder ส่งมา
   - **แผนปฏิบัติงาน (Action Plan)** — ระบุชัดว่า **Founder ต้องทำอะไรเอง** (เช่น ส่งลิงก์กลุ่มเพิ่ม, ให้เคสจริง, ตัดสินใจ Approve ข้อความ) แยกจากสิ่งที่ AI จะทำเองต่อไปโดยไม่ต้องรอ

**ห้าม** ทำเป็นหน้าเว็บหลายหน้าที่ Founder ต้องเข้าไปไล่เช็คทีละ Agent ทีละสถานะ — ต้องเป็นการสนทนา/ผลลัพธ์แบบรวมศูนย์ที่จุดเดียว เหมือนคุยกับผู้จัดการสำนักงานคนเดียวที่ไปสั่งงานลูกน้องมาให้แล้ว

## 1. บริบทธุรกิจที่ AI ทุกตัวต้องรู้ (Business Context — ฝัง Prompt นี้ในทุก Agent)

**CSC คืออะไร:** SaaS ระบบจองคิวออนไลน์สำหรับร้านทำเล็บ/ต่อขนตา/ทำผม/นวด/เสริมสวย เป้าหมาย: ลดงานตอบแชท ลดลูกค้าเท ลดคิวชน ให้ลูกค้าจองเองได้ 24 ชม.

**Stage ปัจจุบัน:** Early Product-Market Fit — มีระบบใช้งานจริง มีร้านจ่ายเงินจริง 3 ร้าน มี Onboarding อัตโนมัติแล้ว **ยังไม่มี** ทีม/ระบบ Support/Sales Funnel/Brand Awareness

**คอขวดจริง (ท่องให้ขึ้นใจ):** ไม่ใช่โค้ด แต่คือ Funnel 4 จุด: **Lead Generation → Sales → Trust → Retention**

**ห้ามแนะนำให้ทำตอนนี้:** POS, Stock, ERP, HR, CRM ใหญ่โต — โฟกัสแค่ Booking / Deposit / Schedule / Customer Flow

**Pain Point ลูกค้า (เรียงตามความสำคัญจากการสัมภาษณ์จริง):**
1. ลูกค้าเท
2. ตอบแชทไม่ทัน
3. จัดคิวยาก
4. เลื่อนนัดบ่อย
5. ทำงานคนเดียว

**จุดแข็งที่ต้องขาย (ไม่ใช่ฟีเจอร์):** ลูกค้าจองเอง / รับมัดจำ / ลดงานตอบแชท / จัดการคิว / เปิดรับจอง 24 ชม.

**เป้าหมายธุรกิจ:** ระยะสั้น 3→10 ร้าน / ระยะกลาง 10→30 ร้าน / ระยะยาว 100 ร้านจนระบบเลี้ยงตัวเอง

**Founder — จุดแข็ง/จุดอ่อนที่ AI ต้องชดเชยให้:**
- แข็ง: เขียนระบบ, พัฒนาฟีเจอร์, แก้ปัญหา, สร้าง Automation
- อ่อน: ไม่ชอบทักลูกค้า, ไม่ชอบขายตรง, ไม่ชอบ Follow-up, หมดพลังเมื่อคุยกับคนเยอะ
- **ดังนั้น เป้าหมายของระบบทั้งหมดคือลดภาระการขาย-ตอบลูกค้า-ซัพพอร์ตให้ Founder มากที่สุด ไม่ใช่ช่วยเขียนโค้ด**

## 2. Agent Roster (6 ตัว — แทนที่ชุด 4 ตัวเดิมทั้งหมด)

| Agent | หน้าที่ | Output ที่ต้องส่งเข้า Supervisor |
|---|---|---|
| **1. Lead Hunter** | วิเคราะห์ข้อมูลกลุ่ม Facebook/คอมเมนต์ที่ Founder แปะมา หา Pain Point ตรงกับ 5 อันดับข้างต้น | รายชื่อ Lead + Pain Point ที่พบ + ระดับความเร่งด่วน |
| **2. Sales Assistant** | ร่างบทเปิดคุย Messenger โดยเปิดด้วยคำถาม Pain Point เช่น "ตอนนี้ร้านมีปัญหาเรื่องอะไรที่สุดครับ" ห้ามขายตรงวันแรก | ข้อความร่าง + เหตุผลที่เลือกมุมนี้ |
| **3. Demo Agent** | เตรียมการตอบคำถามฟีเจอร์/เทียบแพ็กเกจ โดยขาย "จุดแข็ง" ไม่ใช่ฟีเจอร์ยิบย่อย | สรุปประเด็นที่ควร Demo + สิ่งที่ห้ามพูดถึง (POS/CRM ฯลฯ) |
| **4. Onboarding Agent** | ตรวจสอบว่า Flow สอนตั้งค่าร้าน/TOTP/เพิ่มบริการ มีจุดไหนที่ลูกค้าใหม่น่าจะติดขัด | รายการจุดเสี่ยงใน Onboarding |
| **5. Customer Success Agent** | เช็กจากข้อมูลที่มีว่าร้านที่ใช้อยู่มีการจองจริงหรือไม่ ร้านไหนเสี่ยงหลุดหลังเดือนแรก | รายชื่อร้านเสี่ยง + เหตุผล |
| **6. Product Analyst Agent** | จัดกลุ่ม Feedback ที่ได้รับ เสนอ Roadmap แบบย่อ (ต้องเช็คด้วยว่าตรงข้อห้าม POS/CRM หรือไม่ ถ้าตรงให้ปฏิเสธเสนอ) | สรุป Feedback + คำแนะนำ Roadmap (ถ้ามี) |

**Supervisor Agent (ตัวที่ 7 — ตัวกลาง):**
- อ่าน Input ดิบที่ Founder แปะเข้ามา ตัดสินใจว่าควรกระจายงานให้ Agent ไหนบ้าง (ไม่จำเป็นต้องเรียกทุก Agent ทุกครั้ง — เรียกเฉพาะที่เกี่ยวข้องกับข้อมูลที่ได้รับ)
- รวบรวมผลลัพธ์จากทุก Agent ที่ถูกเรียก มาสังเคราะห์เป็นคำตอบเดียว 2 ส่วน: **ข้อมูลสำคัญที่พบ** และ **แผนปฏิบัติงาน**
- ในแผนปฏิบัติงาน ต้องแยกให้ชัดเจนเป็น 2 กลุ่มเสมอ:
  - **สิ่งที่ Founder ต้องทำเอง** (เช่น ให้ลิงก์กลุ่มเพิ่ม, ให้เคสจริงยืนยัน, ตัดสินใจ Approve)
  - **สิ่งที่ AI จะทำต่อเองโดยไม่ต้องรอ** (เช่น ส่ง Follow-up ตามคิว, บันทึกลีดใหม่)
- ถ้า Agent ใดขาดข้อมูล (เช่น Lead Hunter เข้ากลุ่ม Facebook ที่ปิดไม่ได้) **ต้องระบุตรงๆ ในผลสรุปว่าขาดอะไร แล้วขอจาก Founder** แทนการเดาหรือข้ามไปเงียบๆ

## 3. Interaction Model (แก้จากพร็อมเดิม)

- **Input:** ช่องเดียว (chat box หรือ textarea) ให้ Founder แปะข้อมูลดิบเข้ามาได้ทุกรูปแบบ (ข้อความ, โพสต์คัดลอกมา, สรุปบทสนทนา)
- **Processing:** เบื้องหลังเป็น LangGraph ที่ Supervisor เป็น entry point แจกงานให้ Agent ที่เกี่ยวข้องทำงานตามลำดับ/ขนานกัน (ยังคงเดินผ่าน Rate Limiter เดิมทุกครั้งที่ยิง LLM)
- **Output:** ข้อความตอบกลับเดียวในหน้าจอเดียว โครงสร้างตายตัว:
  1. หัวข้อ "ข้อมูลสำคัญที่พบ" — bullet สั้นๆ จากแต่ละ Agent ที่ถูกเรียกใช้งานจริง (ไม่ต้องแสดง Agent ที่ไม่เกี่ยวข้อง)
  2. หัวข้อ "แผนปฏิบัติงาน" — แยก 2 กลุ่มตามข้อ Supervisor ด้านบน
  3. ถ้ามีข้อความร่าง (เช่นจาก Sales Assistant) ที่ต้อง Approve ก่อนส่งจริง ให้แสดงในหน้าเดียวกันนี้พร้อมปุ่ม Approve/Edit/Reject — **ไม่แยกไปหน้า Pending Approvals ต่างหาก**

## 4. สิ่งที่ยังคงไว้จากพร็อมเดิม (ไม่เปลี่ยน)

- Database Schema (leads, api_usage_log, pending_approvals, agent_feedback, research_cache) — ยังใช้ table เดิม แค่ปรับ `agent_name` ให้ตรงกับ Agent 6 ตัวใหม่
- Rate Limiter สำหรับ GitHub Models Token (Token Bucket, Sleep Mode, Request Queue, Research Cache) — ใช้ตามเดิมทุกข้อ ห้ามข้าม
- Research-First — ทุก Agent ที่อ้างข้อมูลการตลาด/เคสลูกค้าต้องมีแหล่งอ้างอิงจริง ห้ามแต่งขึ้น ถ้าไม่มีข้อมูลให้บอก Founder ว่าขาดอะไร (ตรงกับ Concept "ถ้า AI หาข้อมูลจากกลุ่ม Facebook ไม่ได้ Founder จะหามาให้")
- Self-Improvement mechanism — เก็บ feedback จากผล Approve/Reject มาสรุปคำแนะนำ แต่ **ไม่ต้องทำเป็นหน้า "Weekly Insights" แยก** ให้ผนวกเข้าไปในผลสรุปรอบถัดไปแทน (เช่น ถ้า Sales Assistant เคยถูก Reject บ่อยเรื่องโทนภาษา ให้ Supervisor เตือนไว้ในข้อ "ข้อมูลสำคัญที่พบ" ของรอบใหม่)

---

## 5. สิ่งที่ต้องลบ/ไม่ต้องทำจากพร็อมเดิม

- ❌ ไม่ต้องสร้างหน้า Admin Dashboard แบบหลายหน้า (Pending Approvals / Leads Overview / System Health / Weekly Insights / Knowledge Base Manager แยกกัน)
- ❌ ไม่ต้องมี Agent ชุดเดิม 4 ตัว (Lead Scraper, Strategic Closer, Support Guide, Supervisor) — แทนที่ด้วย 6+1 ตัวใหม่ตามข้อ 2
- ❌ ไม่ต้องมีระบบซัพพอร์ตลูกค้าปลายทาง (Agent 3 เดิม) เป็นส่วนแยก — ตอนนี้ยังไม่ถึง Stage ที่ต้องมี Support Team ตามที่ระบุใน Business Context (ยังไม่มีระบบ Support จริง)

---

## 6. Deliverables ที่ต้องการจาก Replit Agent (ฉบับแก้)

1. ปรับ Agent modules จาก 4 ตัวเดิม → 6 ตัวใหม่ + Supervisor ตามข้อ 2 พร้อม prompt ของแต่ละ Agent แยกเป็น config
2. หน้าเว็บเดียว: ช่อง Input รับข้อมูลดิบ + พื้นที่แสดงผลลัพธ์ (Key Findings + Action Plan + Approval inline ถ้ามี)
3. ฝัง Business Context (ข้อ 1) เป็น system prompt ร่วมที่ทุก Agent อ่านเห็นเหมือนกัน เพื่อไม่ให้ Agent ไหนแนะนำ POS/CRM/ERP หรือหลุด Pain Point ที่กำหนดไว้
4. คง Rate Limiter, Research Cache, DB Schema เดิมไว้ ปรับแค่ agent_name ให้ตรง
5. README อธิบายว่า Flow การใช้งานจริงคือ: Founder แปะข้อมูล → รอผลสรุปเดียว → ทำตาม Action Plan ที่ AI ระบุว่าต้องทำเอง
